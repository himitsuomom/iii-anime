## 2. build ループの試算（1 アテンプト = 20 ターン想定）

前提: 各ターンで会話履歴が約 +3k トークン成長（assistant 出力＋tool_result）。
不変接頭辞（system＋Spec/Plan）= 6k。出力 ~2k/ターン。

### 方式A: 接頭辞のみキャッシュ（履歴は毎回フル再送＝素朴）

API はステートレスで毎ターン全 messages を再送するため、履歴は**毎回フル課金**:
- 非キャッシュ入力 ≈ Σ(3k·t, t=1..20) = 3k·210 = **630k** → 630k×$5 = **$3.15**
- キャッシュ読込 ≈ 6k×19 = 114k → ×$0.5 = $0.06
- キャッシュ書込 ≈ 6k → $0.04
- 出力 ≈ 40k → ×$25 = **$1.00**
- **小計 ≈ $4.25 / アテンプト**

### 方式B: マルチターン増分キャッシュ（毎リクエスト最終ブロックに breakpoint）★採用

履歴の既存部分は**キャッシュ読込(0.1x)**になり、新規分のみ通常課金＋書込:
- キャッシュ読込 ≈ Σ(3k·(t-1)) = 3k·190 = 570k → ×$0.5 = **$0.29**
- 非キャッシュ新規入力 ≈ 3k×20 = 60k → ×$5 = $0.30
- キャッシュ書込 ≈ 60k → ×$6.25 = $0.38
- 出力 ≈ 40k → ×$25 = **$1.00**
- **小計 ≈ $1.97 / アテンプト**（**出力が支配項**になる＝ほぼ下限）

> 方式B で約 **半減**。これ以上は「ターン数を減らす」「出力（生成コード量）を減らす」しか効かない
> ＝ **プロンプト改善（過剰実装抑制・無駄 narration 削減）が直接コストに効く**。

## 4. 削減レバー（効果順）

1. **マルチターン増分キャッシュ**（方式B）: 約 2x 削減。最優先で実装。
2. **過剰実装の抑制**（BUILD_SYSTEM_PROMPT §1）: 出力＝支配項を直接削る。
3. **ターン/反復の上限**（`MAX_TURNS`, `max_iterations`）: 上振れの天井を固定。
4. **Task Budget**（beta, Opus 4.8 対応）: ループ総トークンを Claude に自己管理させる。
5. **effort の工程別最適化**: intake=`medium`, design=`high`, build=`xhigh`（品質に効く所だけ高く）。
6. **モデル使い分け**: 安価工程は Haiku/Sonnet（既に反映済）。
7. **Spec/Plan の簡潔化**: 接頭辞を小さく保ち、キャッシュ最小サイズ（Opus 4.8 は 4096 トークン）を意識。
