統合アーキテクチャ設計

EC(couxo9・Python) × Automation Studio(TS) を iii オーケストレーション基盤でモノレポ統合

contract-first / iii backbone / monorepo

方針

1. モノレポ統合 — EC(Python) と Automation Studio(TS) を1リポジトリに同居。

2. iii をバックボーンに — 両者を iii worker 化し、エンジン(ws://localhost:49134)経由の JSON/WebSocket で連携。trigger() 同期 / TriggerAction.Enqueue 非同期 / Void 投げっぱなし。

3. 単一の真実 = packages/contracts — JSON Schema から TS型と Pydantic を生成し両言語で共有。

制約（重要）

・couxo9/EC は private・Python・本セッションのスコープ外で内部実装を参照できない。 よって contract-first 設計とし、EC は薄いアダプタで worker 化（実装が判明しても差分を局所化）。

・EC をスコープに取り込む手順は Phase 0（現状ブロッカー）。

① 商品説明の一括生成（非同期キュー）

sequenceDiagram participant U as 運用者UI(AS) participant AS as automation-studio worker participant E as engine(queue) participant EC as apps/ec worker U->>AS: 対象商品IDリストで一括生成 loop 各商品 AS->>E: trigger(ai::describe-product, Enqueue{queue:"ai-gen"}) E->>AS: 実行（Claude or テンプレ） AS->>E: trigger(catalog::update-description,{id,result}) E->>EC: 説明文を保存 end AS-->>U: 進捗・完了

② 問い合わせ自動応答（state駆動）

sequenceDiagram participant EC as apps/ec participant E as engine participant AS as automation-studio EC->>E: 新規問い合わせ（state trigger 発火） E->>AS: ai::answer-inquiry(inquiry) AS->>E: orders::get(orderId) で文脈取得 E->>EC: 注文情報 EC-->>AS: order AS->>E: 応答下書き（Claude/FAQ）→ EC へ反映 or 人間レビュー

③ KPIダッシュボード: Dashboard.tsx のモック KPIS を orders::stats / inventory::alerts の trigger() 取得に置換。

統合API（関数コントラクト）

関数IDは namespace::function（グローバル一意・worker非依存）。これがサービス間の契約。

提供元関数ID入力 → 出力
apps/eccatalog::get{id} → Product
apps/eccatalog::update-description{id, GeneratedDescription} → {ok}
apps/ecorders::stats{period} → OrderStats
apps/ecinventory::alerts{} → InventoryAlert[]
apps/ecpricing::context{sku} → PricingContext
automation-studioai::describe-productDescribeRequest → GeneratedDescription
automation-studioai::answer-inquiryInquiry → {reply, source}
automation-studioai::suggest-price{sku} → PriceQuote
