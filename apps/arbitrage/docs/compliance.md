# コンプライアンス論点（越境転売）

> ⚠️ 本書は「設計に織り込むべき論点」の整理であり、法的助言ではありません。最終判断は必ず
> **弁護士・税理士・各公式窓口**で確認してください（私は弁護士・税理士ではありません）。
> マスター指示書 §1 / §8 に対応します。Phase 0 ではこれらを**データ構造と運用方針**に
> 反映するに留め、未確定事項を前提とする自動取引は実装しません。

## 1. 古物商許可（古物営業法）

- 中古品を**反復継続して仕入れ・転売**して利益を得る行為は、原則として**古物商許可**
  （営業所所在地を管轄する公安委員会＝警察署経由）が必要になり得ます。本システムは
  「仕入れて売る」が前提のため**該当する可能性が高い**。
- **取引記録（古物台帳）の保存義務**が生じ得ます。いつ・何を・いくらで・誰から/誰へ、を
  記録します。
- **設計への反映**:
  - contract `LedgerEntry`（`packages/contracts` の `kobutsu-ledger` スコープ）が
    取引日時・品名・数量・金額・相手方（氏名/住所/確認方法）・仕入れ元URLを保持。
  - `ledger::record` / `ledger::list` で記録・抽出。月次/年次のCSV輸出は後フェーズ。
- **要確認（§8-2）**: 許可の取得状況、台帳の保存要件（電子保存の可否・保存期間）。

## 2. 各プラットフォームの利用規約（ToS）

- メルカリ・ヤフオク!・ラクマ・magi・SNKRDUNK・Amazon 等の多くは、規約で
  **スクレイピング/自動操作（bot）を禁止または制限**しています。違反は主に
  **規約違反＝アカウント凍結**のリスクです。
- とくに**自動出品・自動値下げ・自動メッセージ**は検知されやすく凍結の典型原因。
- **設計への反映**:
  - **公式API優先**（販売側 eBay は公式 Developer API を主軸）。
  - 国内仕入れ側は公式買い手APIが乏しいため Playwright を**最終手段**とし、
    **低速・少量・人間が見ている前提**（`config/default.yaml` の
    `sourcing.interval_seconds` / `max_items_per_run` で制御）。並列大量アクセスをしない。
  - 出品・取引・購入の「最後のひと押し」は**人間が行う**（human-in-the-loop）。Phase 0 は
    `ListingDraft.mode=dry_run, status=draft` 固定（投稿しない）。
- **要確認（§8-1）**: 対象PFごとの最新ToS、スクレイピング可否・レート、API有無。

## 3. 複数アカウント

- 「カテゴリごとにアカウントを分ける／複数アカウント運用」は、PFによっては**規約違反**で
  まとめて凍結される原因になります。
- **設計への反映**: 拡張は「アカウント数を増やす」前提で作らず、**まず1アカウントで
  規約順守しながら実績を作り、規約上問題ない範囲でのみ拡張**する（§8-3）。

## 4. 税務・輸出

- 継続的な利益は**確定申告**の対象。越境販売は**輸出**にあたり、相手国の関税・消費税
  （VAT/GST）の扱いも絡みます（eBay は国際取引で各国 VAT 等を自動徴収する仕組みあり）。
- **設計への反映**: `LedgerEntry` を申告に使えるCSV/帳票として出力できるようにする
  （月次・年次。輸出は後フェーズ）。`ProfitBreakdown` は手数料・送料・為替を明示控除。
- **要確認（§8-9）**: 申告区分、VAT の取り扱い、必要な帳票様式。

## 5. 為替・利益保全（事故防止）

- **為替バッファ**（§3-6）: 利益計算は実効レートを**悲観側**（`FxRate.effectiveRate =
  rate * (1 - bufferPercent/100)`）に倒し、¥利益フロアを過大評価しない。
- **在庫同期 / 二重販売防止**（§3-5）: 一点物の中古品を扱うため、同一仕入れ品の二重出品を
  `arb-listings` スコープで防止する。仕入れ元で売り切れたら出品を「要取り下げ」にする運用を
  後フェーズで自動化。

## 結論

越境転売は合法的なビジネスです。ただし上記を**設計の前提**に組み込むことで
「凍結されにくく・申告できる・持続する」システムにします。未確定の §8 事項を前提とする
自動出品・自動取引は、確定するまで実装しません（Phase 0 は Dry-run・下書きのみ）。
